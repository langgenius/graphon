# Code Review: 8428e5c

Commit: `8428e5c0ac9559ffbf40076058de9e46a947582b`

Title: `Pass tool node execution id to runtime`

## Inferred Goal

This commit updates `ToolNode.run()` so the node restores or creates its node
execution id before obtaining the tool runtime, then passes that execution id
to the workflow-layer tool runtime adapter.

The commit also attempts to preserve compatibility with older runtime adapters
that do not yet accept the new `node_execution_id` argument.

## Review Scope

- SOLID principles and separation of responsibilities.
- Strong typing and schema usage instead of unstructured maps where relevant.
- Backward compatibility, especially for existing workflow runtime adapters and
  historical tool node data.

## Findings

### P1: Preserve legacy `variable_pool` behavior

File: `src/graphon/nodes/tool/tool_node.py:115`

The commit's stated goal is to pass the node execution id to the runtime, but it
also removes the historical guard that passed `variable_pool=None` for legacy
tool nodes where `version == "1"` and `tool_node_version is None`.

The removed comment explicitly says this branch was kept for backward
compatibility. Always passing `graph_runtime_state.variable_pool` can break
adapters that still use `None` to trigger legacy parameter handling.

Recommended fix: keep the old `variable_pool` selection logic and pass
`node_execution_id` independently.

### P2: Avoid masking runtime `TypeError`

File: `src/graphon/nodes/tool/tool_node.py:259`

If signature inspection says the runtime accepts `node_execution_id`, the code
still catches `TypeError` and retries without the argument when the message
contains `node_execution_id` and `unexpected keyword`.

This can hide a real bug inside a runtime that accepts the parameter but
forwards keyword arguments incorrectly. It may also initialize runtime state
twice.

Recommended fix: separate the compatibility branches. Call with
`node_execution_id` only when the inspected signature advertises support, and
call without it when it does not. Do not fallback after entering an explicitly
compatible runtime.

### P3: Remove unused import

File: `src/graphon/nodes/tool/tool_node.py:4`

`ruff` reports `F401` because `TYPE_CHECKING` is imported but unused.

Recommended fix: remove the unused import before merging.

### P3: Restore top-level spacing

File: `tests/nodes/tool/test_tool_node.py:111`

`ruff` reports `E302` because there is only one blank line before the top-level
`_build_tool_node` function.

Recommended fix: add the missing blank line.

## Strong Typing Notes

This commit does not introduce a new business data map that clearly needs a new
Pydantic schema. The existing `build_file_reference(mapping=...)` protocol still
uses a `Mapping[str, Any]`, but that is pre-existing behavior rather than the
main risk introduced by this commit.

The runtime compatibility code would benefit more from explicit protocol
capabilities or a narrower adapter interface than from additional unstructured
runtime inspection.

## Assessment

Ready to merge: no.

The execution-id goal is valid, and the tests cover the happy path plus one
legacy adapter shape. However, the commit currently includes an unrelated
compatibility regression in `variable_pool` handling and a fallback path that
can mask runtime adapter bugs. Fix P1 and P2 before merging, then clean up the
two lint failures.
