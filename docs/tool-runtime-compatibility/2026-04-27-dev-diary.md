## 1. Feature 目的

本次 feature 的直接目标是让 `ToolNode.run()` 在获取 tool runtime 前先恢复或创建当前
node execution id，并把该 id 通过 `node_execution_id` 传给 workflow 层 runtime adapter。

Dify 上游使用这个 id 的场景是 workflow-as-tool tracing：`DifyToolNodeRuntime` 在 workflow
tool 场景下把外层 `workflow_run_id` 和当前 tool node 的 `node_execution_id` 组合成
`parent_trace_context`，再传给被嵌套调用的 workflow，使 trace provider 能把内外层 workflow
span 关联起来。

同时，本次 feature 不能破坏既有 legacy tool node 参数解析行为。历史 tool node 满足
`node_data.version == "1"` 且 `node_data.tool_node_version is None` 时，Graphon 原本会向
runtime 传 `variable_pool=None`。Dify 的 tool 参数转换逻辑会根据 `variable_pool` 是否存在走
不同解析路径，因此这一行为需要继续保留。

## 2. 实现与迭代过程

最初实现把 `ensure_execution_id()` 前移到获取 runtime 之前，并把返回值作为
`node_execution_id` 传给 runtime。为了照顾可能尚未升级的 runtime adapter，初版还加入了基于
`inspect.signature()` 的兼容分支：adapter 签名支持 `node_execution_id` 时传新参数，否则走旧
调用形态。

随后对初始提交 `8428e5c0ac9559ffbf40076058de9e46a947582b` 做了代码审查，形成中英文审查文档。
审查指出两个问题：

- 初版在传 execution id 的同时误删了 legacy `variable_pool=None` 选择逻辑，可能改变历史 tool
  node 的参数解析路径。
- 初版曾通过捕获 `TypeError` 并文本匹配 `"node_execution_id"` / `"unexpected keyword"` 来
  fallback 到旧调用，这可能掩盖 runtime 内部真实错误。

修复阶段先恢复 `_run()` 中的 legacy `variable_pool` 分支，再把 `_get_tool_runtime()` 的
`variable_pool` 参数类型调整为 `VariablePool | None`。之后删除基于异常文本的 retry，只保留
签名判断分支。

在检查 Dify 上游分支后，发布顺序被明确为：先发布 Graphon，再合入 Dify PR。Dify 的真实接入方
`DifyToolNodeRuntime.get_runtime()` 已经接收 `node_execution_id: str | None = None`。因此
Graphon 不需要长期维护旧 adapter 签名兼容层，最终删除 `_runtime_accepts_node_execution_id()` 和
`inspect.signature()` 逻辑，让 `_get_tool_runtime()` 无条件按新协议传
`node_execution_id`。

测试实现也随之收敛：legacy/current `variable_pool` 两个 happy path 被合并成参数化测试；旧
adapter 兼容测试改成“adapter 必须接收 `node_execution_id`”的协议要求测试。

## 3. 本地 Feature 测试覆盖点

本地 focused 测试在 `tests/nodes/tool/test_tool_node.py` 中覆盖以下行为：

- `ToolNode.run()` 会恢复 runtime state 中已有的 execution id，并传给 runtime 的
  `node_execution_id` 参数。
- legacy tool node，即 `version == "1"` 且 `tool_node_version is None`，仍向 runtime 传
  `variable_pool=None`。
- current tool node，即 `tool_node_version` 不为 `None`，向 runtime 传真实 `VariablePool`。
- runtime adapter 如果不接收 `node_execution_id`，不再被当作兼容 adapter；`Node.run()` 会产生
  失败事件并暴露相关错误。
- 既有 `transform_message()` 行为仍覆盖 text、variable、file、image link 和非法 file payload
  场景，确保本次 runtime 获取逻辑改动没有顺手破坏 tool message 转换测试。
